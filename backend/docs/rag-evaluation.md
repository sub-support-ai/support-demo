# Оценка качества RAG

Этот документ — для двух аудиторий: разработчиков, которые хотят что-то
улучшать в retrieval'е, и админов KB, которые видят промахи на проде.

## Зачем это нужно

Без эвал-набора любая правка ranking — догадка. «Поменяли веса FTS / semantic» —
стало лучше или хуже? «Подняли порог HIGH» — больше промахов или больше шума?
Без числа ответа нет, и каждое изменение становится политическим спором.

Эвал-набор — это набор реалистичных запросов с эталонными правильными ответами.
По нему считаются три метрики:

- **precision@1** — попал ли в первый результат правильный ответ. Это то, что
  видит пользователь — его главная метрика.
- **recall@5** — нашёлся ли правильный ответ хотя бы в top-5. Если recall высокий,
  а precision@1 низкий — нужен reranker.
- **MRR** (mean reciprocal rank) — среднее значение `1 / rank` по всем запросам.
  Чувствительно к падению с rank=1 на rank=2 (с 1.0 до 0.5).

## Где лежит

```
backend/
├── evals/
│   ├── __init__.py
│   ├── metrics.py          # precision@k, recall@k, MRR — чистые функции
│   ├── dataset.py          # YAML-загрузчик + нормализация title'ов
│   └── data/
│       └── baseline.yaml   # эвал-набор; растёт по мере находок в проде
└── tests/
    └── test_rag_evaluation.py  # pytest-регрессия по baseline.yaml
```

## Текущий baseline

На 15 кейсах в SQLite (FTS-only, без semantic):

| Метрика     | Значение | Что значит                              |
|-------------|----------|------------------------------------------|
| precision@1 | 0.80     | 12 из 15 запросов — правильный ответ #1 |
| recall@5    | 0.80     | те же 12 — в top-5                       |
| MRR         | 0.80     | в 80% случаев правильный ответ на месте 1 |
| Misses      | 3 / 15   | три «длинные перефразировки» FTS не тянет |

**Промахи** (это и есть TODO для следующих улучшений):

1. `«впн отвалился, как починить»` → не нашёл `«VPN не подключается»`
   (синонимы / транслит — query rewriting через Mistral подтянет).
2. `«пароль не подходит при входе в почту»` → не нашёл `«Сброс пароля»`
   (контекст `«почта»` сбивает — reranker должен помочь).
3. `«wi-fi есть, но страницы не грузятся»` → не нашёл
   `«Нет интернета на рабочем месте»` (семантика отрицания — это работа
   embedding-модели; на SQLite её нет, на Postgres+pgvector должно стать лучше).

## Как добавить новый кейс

Когда саппорт жалуется «AI не нашёл очевидное» или вы заметили промах в чате —
добавьте кейс в `backend/evals/data/baseline.yaml`:

```yaml
cases:
  - query: "формулировка пользователя слово в слово"
    expected_titles:
      - "Заголовок статьи KB, которую должен был найти RAG"
    notes: "Источник: тикет #427, демо 2026-04, что угодно"
```

Несколько правильных ответов:

```yaml
  - query: "запрос, на который подходят 2 статьи"
    expected_titles:
      - "Первая подходящая статья"
      - "Вторая подходящая статья"
```

После добавления:

```bash
cd backend
py -3.12 -m pytest tests/test_rag_evaluation.py -v
```

Если упало — это новый промах, попавший в эвал-набор. Не понижайте пороги в
тесте, чтобы прошло. Вместо этого:
- Если кейс крайний — пометьте его в `notes`, оставьте, дальше системно
  улучшайте retrieval (query rewriting, reranker).
- Если кейс «должен был работать» — это регрессия в коде, ищите её.

## Когда поднимать пороги

Каждый раз, когда какое-то улучшение системно поднимает метрики — поднимите
пороги в `tests/test_rag_evaluation.py`:

```python
MIN_PRECISION_AT_1 = 0.85  # было 0.75 после reranker'а — фиксируем новый уровень
```

Пороги — это контракт: «качество retrieval'а не должно падать ниже этого». Если
будущая правка ронит метрики, тест громко скажет об этом в PR.

## Что не покрывает

- **Семантический поиск** — на SQLite нет pgvector, в pytest гоняется только
  FTS-fallback. Чтобы получить полную картину, прогоняйте эвал на проде через
  отдельный CLI (TODO — будет в следующих итерациях).
- **Скоринг confidence** — эвал-набор проверяет порядок результатов, не
  decision (`answer / clarify / escalate`). Для этого нужен отдельный набор
  «низкоуверенных» запросов.
- **End-to-end чат** — здесь только retrieval. Какой ответ генерирует
  Mistral по найденной статье — отдельная задача (LLM-as-a-judge или
  ручная разметка).

## Метрики на проде

В `AILog` уже пишется `confidence_score`, `model_version`, `ai_response_time_ms`
(см. Блок 3). На основе их можно следить за тем, как ведёт себя AI в проде:

- Дашборд «Сбои AI за 24ч» (Блок 7) — внешние сбои.
- Feedback-агрегаты на статьях (Блок 6) — `helped_count / not_helped_count /
  not_relevant_count`. `not_relevant > helped` — статья регулярно ловится на
  не свои запросы; кандидат для эвал-набора.

Эвал-набор — нижний рубеж для разработки; продовая телеметрия — верхний для
эксплуатации. Они дополняют, не заменяют друг друга.
