# Материал для аналитика к предзащите

Дата подготовки: 5 мая 2026.

## 1. Коротко о проекте

Проект — демо-система службы поддержки сотрудников компании. Пользователь пишет проблему в чат, система пытается помочь автоматически, а если нужен специалист, собирает контекст и формирует черновик тикета.

Цель проекта: сократить время первичной обработки обращений и улучшить качество тикетов, которые попадают к специалистам.

Основная идея:

- пользователь описывает проблему обычным языком;
- ассистент уточняет недостающие данные;
- система классифицирует обращение по отделу, категории и приоритету;
- создаётся черновик тикета;
- пользователь проверяет и подтверждает отправку;
- тикет назначается подходящему специалисту.

## 2. Для кого система

Основные пользователи:

- сотрудник компании, который столкнулся с проблемой;
- специалист поддержки, который принимает и закрывает тикеты;
- администратор или руководитель поддержки, который смотрит статистику и качество обработки.

Типовые отделы:

- IT;
- HR;
- finance;
- other для обращений вне классификации.

## 3. Что делал аналитик

Аналитик в проекте отвечал за понимание предметной области и описание требований, чтобы разработка не свелась просто к "чату с моделью".

Что было проработано:

1. Описан основной пользовательский сценарий: от вопроса в чате до подтверждённого тикета.
2. Выделены роли пользователей: обычный пользователь, агент поддержки, администратор.
3. Сформулированы обязательные данные для тикета:
   - кто обращается;
   - email заявителя;
   - офис или локация;
   - затронутый объект;
   - описание проблемы;
   - что уже пробовали сделать.
4. Определены статусы тикета и ограничения:
   - черновик нельзя отправлять без обязательного контекста;
   - пользователь не должен сам ставить критический приоритет;
   - критический приоритет должен назначаться системой по признакам аварии, простоя, массовости или риска безопасности;
   - агент не должен закрывать неподтверждённый пользователем черновик.
5. Описана логика эскалации:
   - если ассистент не уверен;
   - если вопрос требует ручных действий;
   - если пользователь сам просит создать заявку;
   - если есть риск безопасности или срочная физическая проблема.
6. Составлена таксономия обращений для классификации:
   - IT: оборудование, ПО, доступы, сеть;
   - HR: зарплата, отпуск, политики, онбординг;
   - finance: счета, расходы, отчётность;
   - other.
7. Подготовлены требования к датасету:
   - диалог;
   - итоговый тикет;
   - категория;
   - отдел;
   - приоритет;
   - предпринятые действия.
8. Сформулированы критерии качества:
   - тикет содержит достаточно данных для агента;
   - категория и отдел соответствуют проблеме;
   - приоритет не завышается из-за эмоций пользователя;
   - пользователь видит и подтверждает черновик перед отправкой.

## 4. Что уже реализовано в проекте

Реализованы основные части демо:

- авторизация и роли пользователей;
- чат пользователя с ассистентом;
- сбор контекста обращения;
- создание черновика тикета из диалога;
- редактирование черновика перед отправкой;
- подтверждение тикета пользователем;
- запрет самостоятельного выбора критического приоритета;
- назначение тикета агенту;
- действия агента: взять в работу, закрыть тикет;
- статистика по тикетам и качеству обработки;
- фоновая обработка ответов ассистента через jobs в Postgres;
- локальный запуск через PowerShell-скрипт;
- датасет для классификации обращений.

## 5. Основной сценарий для демонстрации

Сценарий:

1. Пользователь входит в систему.
2. Открывает чат.
3. Пишет проблему, например:

   ```text
   Порвался провод у рабочего места, срочно, боюсь что может искрить.
   ```

4. Ассистент не должен просто сказать "обратитесь в поддержку".
5. Он должен понять, что это потенциальная физическая проблема, и попросить уточнить:
   - офис;
   - точное место;
   - что именно повреждено;
   - кто заявитель;
   - что уже сделали.
6. После эскалации создаётся черновик тикета.
7. Пользователь проверяет черновик и подтверждает отправку.
8. Тикет уходит в отдел IT и назначается агенту.

Что важно показать:

- черновик не отправляется сразу без подтверждения;
- пользователь видит поля контекста;
- приоритет "критический" нельзя поставить вручную;
- система сама оценивает срочность;
- после подтверждения тикет становится доступен специалисту.

## 6. Какие требования особенно важны

### Функциональные требования

- Пользователь может создать обращение через чат.
- Система сохраняет историю диалога.
- Система может предложить эскалацию в тикет.
- Тикет должен содержать структурированный контекст.
- Пользователь может отредактировать черновик.
- Пользователь подтверждает отправку тикета.
- Агент видит назначенные тикеты.
- Агент может взять тикет в работу и закрыть его.
- Администратор видит сводную статистику.

### Нефункциональные требования

- Ответ модели не должен блокировать интерфейс.
- Если модель отвечает долго, пользователь должен видеть статус обработки.
- Если AI service недоступен, обращение не должно теряться.
- Backend должен валидировать обязательные поля, а не полагаться только на frontend.
- Решения системы должны быть проверяемыми через сохранённые метаданные.

## 7. Данные, которые нужны для качественного тикета

Минимальный контекст:

| Поле | Зачем нужно |
|---|---|
| requester_name | чтобы агент понимал, от кого запрос |
| requester_email | для связи с заявителем |
| office | чтобы понять локацию проблемы |
| affected_item | чтобы понять, что именно затронуто |
| title | краткая суть обращения |
| body | полное описание |
| steps_tried | что пользователь уже пробовал |
| department | куда маршрутизировать |
| priority | насколько срочно |

Главная аналитическая мысль: проблема без контекста не помогает агенту. Поэтому система должна не только распознать тему, но и собрать недостающие сведения.

## 8. Как устроена классификация

Внутренние категории для обучения подробнее, чем внешний API:

- `it_hardware`
- `it_software_install`
- `it_software_error`
- `it_access_grant`
- `it_access_reset`
- `it_network`
- `hr_payroll`
- `hr_leave`
- `hr_policy`
- `hr_onboarding`
- `finance_invoice`
- `finance_expense`
- `finance_report`
- `other`

Для API часть категорий схлопывается:

- `it_software_install` и `it_software_error` -> `it_software`;
- `it_access_grant` и `it_access_reset` -> `it_access`.

Зачем так сделано:

- модели проще учиться на более точных внутренних классах;
- пользователю и backend не нужно видеть лишнюю детализацию;
- аналитика по потокам обращений получается точнее.

## 9. Приоритеты

Приоритеты:

- `низкий` — вопрос, консультация, пожелание;
- `средний` — проблема есть, но работа не полностью заблокирована;
- `высокий` — работа конкретного пользователя существенно заблокирована;
- `критический` — массовая авария, риск безопасности, потеря данных/денег, простой сервиса.

Важное решение: пользователь не может вручную поставить критический приоритет. Это снижает риск завышения срочности. Критический приоритет должен выдаваться системой по признакам влияния на бизнес.

## 10. Датасет

Текущий формат датасета — JSONL: одна строка, один пример.

Пример содержит:

- `combo` — параметры генерации примера;
- `conversation` — диалог пользователя и агента;
- `ticket` — итоговый тикет;
- `labels` — отдел, категория и приоритет.

Файл с ручной базой:

```text
ai/ai-dataset/data/raw/manual_v2/samples.jsonl
```

Пайплайн обработки:

1. `judge.py` — проверка качества и меток;
2. `dedup.py` — удаление похожих дублей;
3. `build_splits.py` — разбиение на train/val/test.

Команды:

```powershell
cd C:\code\support-demo\ai\ai-dataset
.\.venv\Scripts\Activate.ps1

python scripts\judge.py --run-id manual_v2 --concurrency 4
python scripts\dedup.py --run-id manual_v2 --threshold 0.92
python scripts\build_splits.py --run-id manual_v2
```

## 11. Как объяснить фоновую обработку

Раньше запрос в чат мог ждать ответа модели слишком долго. Это плохо для интерфейса: пользователь мог уйти на другую вкладку или получить timeout.

Сейчас используется очередь jobs в Postgres:

1. пользователь отправляет сообщение;
2. backend сохраняет сообщение;
3. backend создаёт задачу в таблице `ai_jobs`;
4. frontend сразу показывает, что ответ обрабатывается;
5. отдельный worker забирает задачу;
6. worker вызывает AI service;
7. ответ сохраняется в историю диалога;
8. frontend polling-ом подхватывает готовый ответ.

Почему пока Postgres jobs, а не Redis:

- меньше инфраструктуры для учебного проекта;
- проще запускать локально;
- достаточно для демо и небольшой нагрузки;
- позже можно заменить очередь на Redis, не переписывая бизнес-логику.

## 13. Возможные вопросы и ответы

**Почему нельзя сразу создавать тикет из первого сообщения?**

Потому что первое сообщение часто неполное. Агенту нужно знать локацию, объект проблемы, заявителя и уже предпринятые действия. Без этого тикет будет плохого качества.

**Почему пользователь не может выбрать критический приоритет?**

Потому что пользователи часто завышают срочность. Критический приоритет должен зависеть от бизнес-влияния: массовость, простой, безопасность, потеря данных или денег.

**Зачем нужен черновик тикета?**

Чтобы пользователь проверил данные перед отправкой. Это снижает риск неверного заявителя, офиса или описания.

**Почему используется датасет?**

Чтобы классификация была не только на промптах, но и имела проверяемую обучающую базу: диалог, тикет, категория, отдел, приоритет.

**Почему используется очередь jobs?**

Потому что ответ модели может занимать десятки секунд. Очередь позволяет не блокировать интерфейс и продолжать обработку в фоне.

**Что будет, если AI service недоступен?**

Сообщение пользователя сохраняется, система может предложить эскалацию, обращение не теряется.

## 15. Что аналитику нужно помнить

Главный акцент: система решает не задачу "ответить в чате", а задачу "собрать качественное обращение и правильно передать его специалисту".

Три ключевые мысли:

1. Без контекста тикет бесполезен для агента.
2. Приоритет должен определяться влиянием на бизнес, а не эмоциями пользователя.
3. Пользователь должен подтверждать черновик перед отправкой.
